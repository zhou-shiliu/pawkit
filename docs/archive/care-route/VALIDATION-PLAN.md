# Pawkit Validation Plan

## 为什么现在先做验证

Pawkit 当前最大的风险不是“猫咪功能还不够多”，而是“用户会不会真的因为它而形成一点点照顾习惯”。  
所以这一轮不继续堆复杂玩法，而是先验证下面三件事：

- 用户会不会注意到猫咪的需求提示。
- 用户会不会愿意通过菜单栏完成一次照料。
- 用户会不会在多次提示里持续响应，而不是只在第一眼觉得新鲜。

## 当前本地记录的事件

当前版本会在本地记录这些信息：

- 应用启动次数
- 需求提示开始与结束
- 用户是否在提示期间打开菜单栏
- `喂食 / 加水 / 陪玩` 的执行次数
- 每个提示从出现到首次被响应的大致耗时
- 是否出现了“当前需要喝水，但用户做了别的动作”这类偏行动作

这些数据只保存在本地，不会发到外部服务。

## 怎么看报告

- 从菜单栏打开 `打开验证报告`
- 报告会输出一份本地 Markdown 摘要和一份 JSON 快照
- 用它来判断当前版本更像“用户真会照顾的陪伴物”，还是“只看一眼的新奇玩具”

## 这一轮想验证的假设

### 假设 1：提示是能驱动行为的

如果提示出现后，用户会在一段时间内打开菜单栏并执行对应动作，说明这条闭环成立。

优先看这几个信号：

- `First-response rate` 不持续为 0
- `Tray opens while need active` 持续增长
- `Avg first response latency` 不至于长时间失控

### 假设 2：菜单栏是合适的照料入口

如果大多数照料动作都来自菜单栏，而不是只存在于调试或渲染层调用，说明“被动看猫 + 主动从菜单栏照顾”是成立的。

优先看这几个信号：

- `菜单栏触发` 明显高于其他来源
- 用户在需求出现后会主动打开菜单栏，而不是完全无视

### 假设 3：用户愿意重复照顾，而不是只响应一次

如果提示和响应能在同一个用户的后续会话里反复出现，说明这个产品开始有“陪伴习惯”的苗头。

优先看这几个信号：

- `Sessions` 持续增长
- `Total prompts` 和 `promptResponseCount` 一起增长
- `Ignored prompts` 不要无限高于 `promptResponseCount`

## 早期成功信号

这不是正式 KPI，只是下一阶段前的早期判断线：

- 前 3 到 5 个试用者里，至少有人会在同一天内响应多次提示
- 紧急提示的首次响应耗时，明显快于轻提醒
- 有人愿意连续保留应用 2 到 3 天，而不是当天就关掉
- 主观反馈里，用户觉得它“会提醒我照顾一下”，而不是“只是屏幕上多了个装饰”

## 如果验证结果不好，下一步怎么收敛

如果本地报告显示提示很多、响应很少，不应该继续堆玩法，而应该优先处理下面的问题：

- 提示是否太弱，用户根本没注意到
- 提示是否太频繁，用户直接忽略
- 菜单栏入口是否太远，响应成本太高
- 猫咪的反馈是否不够有情绪，用户缺少照顾动机

结论先来自行为，再决定下一轮做提醒节奏、反馈强化，还是入口形态调整。
